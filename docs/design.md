## Problem Statement

~50-100 family DVDs originally recorded on VHS have been digitized (VHS → DVD → .iso → .mkv). The canonical store is .iso files, with .mkv copies already extracted. The videos are named by date range. Jellyfin was tried as a viewer but failed for non-technical family members (grandparents) because the UI has too many screens, options, and settings. The goal is a dead-simple web app where remote family members open a link, see videos organized by time, and press play. Nothing else.

## What Makes This Cool

A horizontal **time machine** interface. Instead of folders, grids, or library UIs, the landing page is a single draggable timeline spanning the full date range of the collection (e.g., 1989–2004). Drag to a year, videos from that era appear below. One interaction model, zero navigation. Grandma drags to "around when Sarah was born" and hits play.

The future vision makes this even better: scene-split the long DVDs into moments, run face recognition, and let the same timeline filter by person. "Show me Sarah growing up" as a single gesture. Same architecture, richer metadata.

## Constraints

- Must work on any device with a browser (iPad, phone, laptop). Smart TV browsers are best-effort — the draggable scrubber requires touch/mouse, though keyboard arrow keys can provide a D-pad fallback.
- Must be grandparent-proof: no accounts, no settings, no confusing UI
- Videos are VHS-era, so 4:3 aspect ratio (not 16:9)
- Hosted on user's existing home k8s cluster, accessed remotely
- .mkv files need transcoding to browser-playable mp4/h264
- Collection is finite (~50-100 DVDs), not a growing library

## Premises

1. **One-time batch transcode** from .mkv to .mp4 (h264+AAC) via ffmpeg. No on-the-fly transcoding.
2. **Custom simple web app** over any existing media server. The problem is UI complexity, not missing features.
3. **Minimal auth** — nginx `auth_basic` with a single shared username/password (e.g., "family" / passphrase). This is server-side enforcement, so video files are actually gated, not just cosmetically hidden. Browsers remember basic auth credentials, so grandparents enter it once. For a friendlier UX, a tiny auth sidecar with `auth_request` could present a custom passphrase page instead of the browser's native dialog, but `auth_basic` works for V1.
4. **App serves transcoded .mp4 files.** The pipeline is separate from the app. The manifest may reference source filenames for traceability, but the app does not depend on access to source files.
5. **Start with date-based organization.** Architect the data model (manifest.json) so scene detection, face recognition, and per-person timelines can be layered on later without a rewrite.

## Approaches Considered

### Approach A: Simple File Listing UI
Same architecture as C (nginx + manifest.json + static files), but with a straightforward grid/list of video cards instead of the timeline interaction. Sort by date, click to play.
- **Effort:** S | **Risk:** Low
- **Pros:** Simplest possible frontend, fastest to build, nothing clever to break
- **Cons:** Just a file listing — functional but not delightful, harder to browse by era

### Approach B: Minimal Backend + SQLite
Small backend (Node/Go/Python) serving an API from SQLite. Handles transcoding pipeline, generates thumbnails, serves a frontend.
- **Effort:** M | **Risk:** Medium
- **Pros:** Database enables future features naturally, could add admin page, API-first for future mobile app
- **Cons:** More moving parts, over-engineered for V1 "browse and play"

### Approach C: Timeline SPA (chosen)
The time machine concept. Single HTML page with draggable year scrubber, video cards below. Static manifest.json for data. One-time batch pipeline for transcoding + thumbnail generation. Cookie-based passphrase gate. Single nginx container.
- **Effort:** S | **Risk:** Low
- **Pros:** The interaction model IS the product, genuinely delightful, same simplicity as A with a distinctive UX, manifest extends for future features
- **Cons:** Timeline quality depends on accurate date parsing from filenames, hover-to-preview adds frontend complexity

## Recommended Approach

**Approach C: Timeline SPA.** Same effort and simplicity as Approach A, but the timeline concept turns a boring file listing into something that makes people say "whoa." The manifest.json data model supports future scene detection and face recognition without a rewrite.

### Filename Convention & Date Parsing

Source .mkv files are named by date range. Expected patterns:

- `1994-12.mkv` → single month
- `1994-12_1995-01.mkv` → date range (Dec 1994 to Jan 1995)
- `1994-12_christmas.mkv` → month with optional label after underscore

The pipeline script parses `YYYY-MM` from the start of the filename. If a second `YYYY-MM` pattern appears (separated by underscore), it becomes the end date. Any remaining text after the dates becomes the title hint.

### manifest.json schema (V1)

```json
{
  "title": "Family Videos",
  "dateRange": { "start": "1989", "end": "2004" },
  "videos": [
    {
      "id": "christmas-1994",
      "title": "Christmas Morning",
      "dateStart": "1994-12",
      "dateEnd": "1994-12",
      "duration": 6135,
      "file": "videos/christmas-1994.mp4",
      "thumbnail": "thumbs/christmas-1994.jpg",
      "sprite": "sprites/christmas-1994.jpg",
      "spriteColumns": 10,
      "spriteInterval": 30,
      "sourceFile": "1994-12_christmas.mkv"
    }
  ]
}
```

Future extensions (no schema break):
```json
{
  "scenes": [
    { "start": 0, "end": 120, "thumbnail": "...", "faces": ["sarah", "grandma"] }
  ],
  "faces": ["sarah", "grandma", "dad"]
}
```

### Pipeline script

A single shell/Python script that:
1. Walks a source directory of .mkv files
2. Parses date ranges from filenames (see Filename Convention section above). The `id` is a slugified version of the filename stem (e.g., `1994-12_christmas.mkv` → `1994-12-christmas`).
3. Transcodes each to .mp4: h264 CRF 23, 480p (native VHS resolution), AAC 128kbps. These defaults balance quality preservation with streaming speed for grandparents on typical home internet.
4. Extracts a poster thumbnail (frame at 10% of duration, 640x480 jpg)
5. Generates a thumbnail sprite sheet: 160x120 per frame, 10 columns, one frame every 30 seconds. Frontend maps hover X-position to sprite grid coordinates for scrub preview.
6. Writes manifest.json

### Frontend

Single-page app, vanilla JS with no build step (simplest possible, no bundler, no dev server, just index.html + app.js):
- **Passphrase gate:** Cookie-based. Enter once, remembered for 30 days.
- **Timeline scrubber:** Sticky at top. Draggable handle. Tick marks where videos exist. Big year display. **Date range filtering:** a video appears for all years it spans (a video dated Dec 1994–Jan 1995 shows when the scrubber is on either 1994 or 1995).
- **Video grid:** Cards with poster thumbnails, titles, dates, durations. Hover shows sprite sheet scrub.
- **Player overlay:** Dark background, native `<video>` element, 4:3 aspect ratio, close button. No custom controls — browser native controls are already grandparent-tested.
- **Mobile:** Single-column grid, tap instead of hover.

## Open Questions

1. **Video titles:** The files are named by date range. Are there human-readable titles for each DVD, or will titles need to be manually added to the manifest? (Could start with dates and add titles incrementally.)
2. **Transcode quality defaults:** CRF 23, 480p, AAC 128kbps are specified as starting defaults. Confirm these are acceptable, or adjust if grandparents are on particularly slow connections (lower CRF = bigger files = more buffering).
3. **HTTPS/domain:** Is there already a domain + TLS cert for the k8s cluster, or does that need setup?
4. **Storage size:** ~50-100 DVDs at ~4-8GB each as .mkv. Transcoded .mp4 at VHS quality could be 1-2GB each. Total ~50-200GB. Is that fine on the k8s cluster storage?
5. **Remote access mechanism:** The k8s cluster needs to be reachable from the internet for grandparents to use it. Is there an existing ingress with a stable domain and TLS (e.g., Cloudflare Tunnel, Tailscale, dynamic DNS)?
6. **Filename format confirmation:** The pipeline assumes filenames like `1994-12.mkv` or `1994-12_1995-01.mkv`. Does the actual naming convention match this? Provide a few real filenames so the parser can be exact.

## Success Criteria

- Grandparent can open a link, enter the family passphrase, and watch a video within 30 seconds of first visit
- Works on iPad Safari, Chrome on Android, and desktop browsers
- Timeline scrubber is the only navigation needed — no menus, no search, no settings
- Videos play without buffering on a typical home internet connection (nginx serves static files with HTTP range request support by default, enabling seeking and resume in large video files)
- Adding a new video is: run the pipeline script, redeploy

## Distribution Plan

- Containerized as a single Docker image (nginx + static files)
- Deployed to user's existing k8s cluster
- Dockerfile in the repo for reproducible builds
- Pipeline script runs locally against the .mkv source files, outputs go into the container image (or a mounted volume)
